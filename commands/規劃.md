# /規劃 — 藍圖管理

管理定義團隊工作流的藍圖檔案。

## 子指令

解析 `$ARGUMENTS` 判斷子指令。沒有指定子指令時，預設執行 `list`。

---

### `/規劃 list`

1. 掃描 `blueprints/` 資料夾中所有 `.md` 檔案。
2. 主列表排除 `_template.md`（但提及它是模板）。
3. 每個藍圖讀取 `## Meta` 區塊，擷取 `name`、`triggers`、`description`。
4. 以表格呈現：

```
可用藍圖：

| 名稱 | 觸發詞 | 說明 |
|------|--------|------|
| ... | ... | ... |

模板：blueprints/_template.md
```

---

### `/規劃 new <名稱>`

1. 檢查 `blueprints/<名稱>.md` 是否已存在。存在的話警告並詢問是否覆寫。
2. 讀取 `blueprints/_template.md`。
3. 複製模板到 `blueprints/<名稱>.md`。
4. 把所有 `{name}` 替換成指定的名稱。
5. 開啟新檔案並解說每個區塊要填什麼：
   - **Meta**：設定觸發詞和說明
   - **Agents**：定義 Task Agents（只寫角色，任務由 Conductor 在執行時指派）
   - **Flow**：設定執行順序
   - **Roles Memory**：選填的記憶檔案對應
   - **Checklist**：Checker 的品質關卡
   - **On Fail**：重試設定
6. 問使用者要現在互動填寫，還是之後自己編輯。

---

### `/規劃 edit <名稱>`

1. 檢查 `blueprints/<名稱>.md` 是否存在。
2. 找不到的話，執行 `list` 並請使用者挑選。
3. 讀取並顯示目前內容。
4. 問使用者想改什麼。
5. 用 Edit tool 修改。
6. 顯示更新後的檔案。

---

### `/規劃 show <名稱>`

1. 檢查 `blueprints/<名稱>.md` 是否存在。
2. 找不到的話，執行 `list` 並請使用者挑選。
3. 讀取並顯示完整藍圖內容。
4. 附上簡短解讀：
   - 會派遣幾個 Agent
   - 流程長什麼樣子
   - Checklist 檢查什麼

---

## 錯誤處理

| 狀況 | 處理方式 |
|------|----------|
| 沒有 blueprints/ 資料夾 | 建立它並通知使用者 |
| 不認識的子指令 | 顯示可用的子指令 |
| 名稱包含不合法字元 | 警告並建議合法的名稱（小寫、連字號、不含空格） |
