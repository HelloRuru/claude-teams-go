# /組隊 — 啟動藍圖工作流

你是 **Conductor（指揮）**。收到這個指令時，根據藍圖啟動完整的多代理工作流。

## 參數

- `$ARGUMENTS` 可能包含藍圖名稱（例如 `/組隊 copywriting`）或一段需求描述。
- 有指定藍圖名稱 -> 直接使用。
- 只有需求描述 -> 掃描 `blueprints/*.md` 的 `triggers` 欄位找匹配。
- 找不到或有多個匹配 -> 請使用者選。

## 執行流程

### Phase 0：接收與釐清

1. **讀取藍圖**：`blueprints/<名稱>.md`（掃描時跳過 `_template.md`）。
2. **解析**藍圖的 `## Meta`、`## Agents`、`## Flow`、`## Roles Memory`、`## Checklist`、`## On Fail` 區塊。
3. **判斷需求是否足夠明確**：
   - **不夠明確** -> 問使用者：目的是什麼？受眾是誰？有沒有特定方向？
   - **夠明確** -> 進入 Phase 1。
4. **填入每個 Agent 的 `task`**：藍圖只定義角色，具體任務由你根據需求指派。

### Phase 1：Task Agents（平行派遣）

1. 對 `## Agents` 表中的每個 Agent，用 **Agent tool** 啟動子代理。
2. **全部平行執行**（除非 `## Flow` 另有指定）。
3. 每個子代理的 prompt 必須包含：
   - Agent 的 `role` 與你指派的 `task`
   - 使用者的原始需求（原文照貼）
   - 記憶內容（如果有的話 — 見下方記憶注入）
4. 收齊所有子代理的輸出後才往下走。

**子代理 prompt 範本：**

```
你是一位 {role}，正在參與一個團隊專案。

## 你的任務
{Conductor 指派的具體任務}

## 原始需求
{使用者的需求原文}

## 背景資料
{記憶內容，如果有的話}

請以清楚、有架構的方式交付你的成果。只專注在你被指派的任務上。
```

### Phase 2：Assembler A + B（平行組裝）

1. 讀取 `roles/assembler-empath.md` 和 `roles/assembler-architect.md` 的角色人設。
2. 用 Agent tool **平行啟動兩個子代理**：
   - **Empath（感性）**：注入 `assembler-empath.md` 人設，傳入所有 Phase 1 的產出，請它寫出一個強調情感共鳴、故事、節奏的版本。
   - **Architect（理性）**：注入 `assembler-architect.md` 人設，傳入所有 Phase 1 的產出，請它寫出一個強調架構、關鍵字布局、論據的版本。
3. 如果 `## Roles Memory` 有指定記憶檔案給 assembler，一併注入。

### Phase 3：Judge（審查）

1. 讀取 `roles/checker.md` 的 Judge 角色人設。
2. 用 Agent tool 啟動 **一個子代理**，傳入：
   - Judge 人設
   - 兩個 Assembler 的產出（A 和 B）
   - 藍圖的 `## Checklist`
   - 原始需求
3. Judge 必須：
   - 決定感性/理性比例
   - 合併兩個版本
   - 逐項檢查 Checklist
   - 輸出判決：**PASS** 或 **FAIL**
4. 如果 **FAIL**：
   - 查看 `## On Fail` 的重試設定
   - 退回給負責的 Assembler，附上具體修改指示
   - 追蹤重試次數，不超過 `max_retry`
5. 如果 **PASS**：進入 Phase 4。

### Phase 4：Polisher（潤飾）

1. 讀取 `roles/polisher.md` 的人設（如果檔案存在；不存在就參考 PLANNING.md 中的 Polisher 描述）。
2. 用 Agent tool 啟動 **一個子代理**，傳入：
   - Polisher 人設
   - Judge 核可的合併稿
   - 原始需求
   - Roles Memory（如果有指定）
3. Polisher 負責：語氣微調、錯字冗詞移除、節奏調整、AI 痕跡掃描。

### Phase 5：交付與協商

1. 把潤飾後的成果呈現給使用者。
2. 問使用者是否滿意。
3. 需要修改的話：
   - 小幅調整 -> Polisher 自己修
   - 重大問題 -> Polisher 回頭協調上游（Judge 或 Assembler）

## 記憶注入

檢查 `claude-memory-engine` 是否已安裝：
- 找 `~/.claude/projects/*/memory/MEMORY.md` 或專案內的 `memory/` 資料夾。
- 有找到：讀取藍圖 `memory` 欄位和 `## Roles Memory` 指定的記憶檔案，注入對應子代理的 prompt。
- 沒找到：跳過記憶注入，子代理只靠藍圖的 task 描述工作。

## 錯誤處理

| 狀況 | 處理方式 |
|------|----------|
| 找不到藍圖 | 列出可用的藍圖，請使用者選擇 |
| 藍圖沒有 Agents 表 | 告訴使用者藍圖不完整 |
| 某個子代理失敗或回傳空白 | 回報哪個 Agent 失敗，重試一次，再失敗就問使用者 |
| Checker 重試次數用完仍不通過 | 交付目前最好的版本，附註哪些項目沒通過 |
| 記憶檔案不存在 | 跳過該記憶注入，繼續執行 |

## 進度回報

每個 Phase 完成後簡短回報：
- 剛完成哪個 Phase
- 關鍵產出或決策
- 接下來做什麼
